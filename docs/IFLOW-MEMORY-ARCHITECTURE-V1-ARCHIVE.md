## 一、核心发现

### 1.1 记忆类型理论（认知心理学 + AI Agent 研究）

| 记忆类型 | 持续时间 | AI Agent 应用 | iFlow 对应 |
|---------|---------|--------------|-----------|
| **工作记忆** | 秒-分钟 | 当前推理步骤 | 内存管理 |
| **短期记忆** | 分钟-小时 | 当前对话上下文 | 会话历史 |
| **情景记忆** | 长期 | 历史事件、项目记录 | `.iflow/history/` |
| **语义记忆** | 永久 | 事实、知识、最佳实践 | `docs/learning/` |
| **程序性记忆** | 永久 | 技能、工作流程 | `.iflow/skills/` |

### 1.2 行业标准架构模式（收敛性发现）

三个独立高价值项目（Manus $2B 收购、OpenClaw 145K+ stars、Claude Code）不约而同采用了 **文件式记忆架构**：

```
共同模式：身份层 → 项目记忆层 → 任务记忆层 → 知识库层
```

### 1.3 文件大小最佳实践（社区共识）

| 层级 | 建议大小 | Token 限制 |
|------|---------|-----------|
| 全局层 | 50-100 行 | < 1000 tokens |
| 工作区层 | 100-150 行 | < 1500 tokens |
| 项目层 | 50-100 行 | < 2000 tokens |
| 模块层 | 不限 | 按需加载 |

### 1.4 Letta 基准测试关键发现

> 简单的文件存储方案在 LoCoMo 基准测试中达到 74.0% 准确率，超过了复杂的向量数据库方案。

**结论**：对于大多数场景，精心设计的文件存储方案已经足够。

## 三、层级记忆内容清单

### 3.1 全局层应保存内容

| 类别 | 内容 | 原因 |
|------|------|------|
| **用户身份** | 称呼、角色、联系方式、GitHub | 个性化交互 |
| **iFlow 能力速查** | 核心命令、工具白名单、执行模式 | 快速查询，避免重复解释 |
| **AI 协作要求** | 称呼约定、思考原则、验证要求 | 统一交互风格 |
| **用户决策偏好** | 简洁优先、开源工具优先、避免过度设计 | 影响 AI 决策方向 |
| **错误模式预防** | 常见错误类型和预防策略 | 防止重复犯错 |
| **知识触发器** | 关键词 → 文档映射 | 按需加载深层知识 |

### 3.2 工作区层应保存内容

| 类别 | 内容 | 原因 |
|------|------|------|
| **工作区概述** | 核心定位、特色 | 快速理解工作区 |
| **项目索引** | 项目列表、技术栈、说明 | 快速导航 |
| **共享资源** | 技能目录、学习资源、面试资料 | 统一资源入口 |
| **MCP 配置索引** | 常用 MCP 列表和用途 | 快速找到合适工具 |
| **工作区偏好** | 代码风格、Git 规范 | 统一开发规范 |
| **当前状态** | 活跃项目、当前分支、待办事项 | 跨会话连续性 |

### 3.3 项目层应保存内容

| 类别 | 内容 | 原因 |
|------|------|------|
| **项目概述** | 一句话定位 + 核心特色 | 快速理解项目 |
| **技术栈摘要** | 表格形式，核心依赖 | 快速参考 |
| **开发命令** | 常用命令（< 10 个） | 快速执行 |
| **项目约束** | 关键规则（< 10 条） | 避免错误 |
| **注意事项** | iFlow 容易出错的地方 | 预防问题 |
| **详细模块** | @ 导入指向 | 按需加载 |

### 3.4 模块层应保存内容（按需加载）

| 类别 | 文件 | 内容 |
|------|------|------|
| **项目结构** | `@.iflow/project-structure.md` | 详细目录结构 |
| **技术文档** | `@.iflow/tech-stack.md` | 技术选型详情 |
| **API 文档** | `@.iflow/api-reference.md` | API 接口说明 |
| **配置参考** | `@.iflow/configuration.md` | 配置文件说明 |
| **性能优化** | `@.iflow/performance.md` | 性能优化策略 |
| **测试策略** | `@.iflow/testing.md` | 测试方法说明 |

## 四、不应该保存的内容

根据 Claude Code 社区最佳实践：

| 类别 | 原因 |
|------|------|
| 标准编码规范 | iFlow 已知 TypeScript/Vue 标准 |
| 可从代码推断的信息 | 浪费上下文 |
| 频繁变化的内容 | 维护成本高 |
| 完整的文档 | 使用 @import 按需加载 |
| 冗余重复信息 | 多层重复浪费 token |
| 过时的事件记录 | 应归档或删除 |

## 六、Memory Bank 参考架构

来自 Cline/OpenClaw 的成功模式：

```
memory-bank/
├── projectbrief.md      # 项目简报（真理源）
├── activeContext.md     # 活动上下文（每次更新）
├── progress.md          # 进度追踪（每次更新）
├── systemPatterns.md    # 系统模式
├── techContext.md       # 技术上下文
└── decisionLog.md       # 决策日志
```

**关键文件**：
- `activeContext.md`：每次会话开始读取，结束时更新
- `progress.md`：记录任务完成情况
- `decisionLog.md`：记录重要决策及其理由

## 九、核心结论

### 除了 iFlow 自身能力，还应保存到层级记忆的内容：

1. **用户决策偏好** - 影响 AI 的决策方向
2. **错误模式预防清单** - 防止重复犯错
3. **知识触发器** - 关键词到文档的映射
4. **MCP 配置索引** - 快速找到合适工具
5. **当前状态追踪** - 跨会话连续性
6. **决策日志** - 记录重要决策及其理由
7. **进度追踪** - 任务完成情况
8. **iFlow 容易出错的地方** - 项目特定注意事项

### 记忆内容黄金法则：

1. **简洁性** - 每一行都要有价值
2. **准确性** - 基于事实，不是猜测
3. **时效性** - 定期更新，删除过时内容
4. **针对性** - 记录平台不知道或容易出错的内容
