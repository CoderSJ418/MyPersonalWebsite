## ⚠️ 关键发现：页面样式是重灾区

### 问题诊断

经过多代理并行深度分析，发现以下核心问题：

| 问题等级 | 问题描述 | 影响 |
|----------|----------|------|
| 🔴 **P0** | **双风格混合** (Aurora + Pixel) | 用户困惑、视觉不一致、认知负担增加 |
| 🔴 **P0** | **页面样式问题严重** | 影响用户体验和专业形象 |
| 🟠 **P1** | **功能 Bug 众多** | 核心功能不可用或异常 |
| 🟠 **P1** | **测试覆盖率 0.38%** | 无法保证代码质量 |

### 双风格混合问题（研究支撑）

根据 UX 设计一致性研究（UX Studio, MockFlow 2026）：

> **"一致性是用户体验的基础。当设计不一致时，用户会感到困惑和迷失。"**
> 
> **"混合多种设计风格会导致：**
> - 用户认知负担增加
> - 品牌信任度降低
> - 学习曲线变陡
> - 跳出率上升"

**本项目双风格混合的具体表现**：
- `HeroSection.vue` 使用 Aurora 渐变风格
- `Pixel组件` 使用复古像素风格
- 两种风格在同一页面混合使用
- 用户无法形成一致的视觉认知

### 解决方案：统一为单一设计风格

**决策：采用 Aurora 风格作为唯一设计语言**

理由：
1. Aurora 风格更现代、更适合个人作品集
2. 像素风格与整体设计语言冲突
3. 单一风格降低维护成本
4. 符合 2025-2026 设计趋势

**执行策略**：
- 保留 Pixel 组件代码（备用）
- 首页和主要页面统一使用 Aurora 风格
- PixelDemo 页面作为独立展示保留

## 一、AI 边界与防幻觉机制

### 1.1 禁止事项（绝对边界）

| # | 禁止行为 | 原因 |
|---|----------|------|
| 1 | **禁止猜测** | 不确定必须验证，不能凭记忆断言 |
| 2 | **禁止假设** | 不假设文件存在、不假设代码行为 |
| 3 | **禁止跳过验证** | 每步修改后必须运行 lint/build/test |
| 4 | **禁止删除前不检查** | 删除文件必须先搜索引用 |
| 5 | **禁止重复犯错** | 同类错误超过2次必须暂停反思 |
| 6 | **禁止过度设计** | 只做必要的事，不画蛇添足 |
| 7 | **禁止创建不必要文件** | 除非绝对必要，不创建新文件 |

### 1.2 必须事项（强制行为）

| # | 必须行为 | 执行时机 |
|---|----------|----------|
| 1 | **先读取再修改** | 修改任何文件前先 read_file |
| 2 | **先搜索再删除** | 删除前用 glob/search_file_content 搜索引用 |
| 3 | **修改后验证** | 每次 modify 后运行 npm run lint |
| 4 | **构建验证** | 每阶段结束运行 npm run build |
| 5 | **测试验证** | 功能修改后运行 npm run test:run |
| 6 | **记录决策** | 重要决策记录到文档 |
| 7 | **50%上下文压缩** | 上下文超过50%立即压缩 |

### 1.3 验证清单

每阶段完成后必须执行：

```bash
# 代码质量验证
npm run lint          # ESLint 检查
npm run build         # 构建验证
npm run test:run      # 测试验证

# 类型验证
npx tsc --noEmit      # TypeScript 类型检查
```

## 九、执行步骤详解

### 步骤 1：启用 BMAD UX 专家进行页面样式全面检测

```bash
# 使用 bmad-ux-expert 进行检测
task(subagent_type="general-purpose", prompt="BMAD UX 专家检测任务")
```

**检测范围**：
- 首页 (Home.vue)
- 项目页 (Projects.vue, ProjectDetail.vue)
- 博客页 (Blog.vue, BlogDetail.vue)
- 关于页 (About.vue, Education.vue, Experience.vue, Skills.vue)
- 联系页 (Contact.vue)

**检测维度**：
1. 视觉一致性（颜色、字体、间距、圆角、阴影）
2. 响应式完整性（6 个断点）
3. 暗黑模式完整性
4. 可访问性（ARIA、键盘导航、对比度）
5. 动画流畅度

### 步骤 2：输出问题清单并排序

每个页面输出：
- 问题截图
- 问题描述
- 严重程度 (P0/P1/P2)
- 修复建议

### 步骤 3：按优先级修复

修复顺序：P0 → P1 → P2

每个修复后：
1. 运行 `npm run lint` 检查代码
2. 运行 `npm run build` 检查构建
3. 截图对比验收

### 步骤 4：统一设计风格

**具体操作**：

1. **确定主色调**
   - 主色：Aurora Cyan (#00FFFF)
   - 辅助色：Aurora Purple (#8B00FF)
   - 强调色：Electric Blue (#0080FF)

2. **确定字体层级**
   - 标题：Inter (font-display)
   - 正文：Inter (font-primary)
   - 代码：JetBrains Mono (font-mono)

3. **确定间距系统**
   - 使用 Tailwind 默认间距
   - 8px 基础单位

4. **确定动画规范**
   - 入场动画：600ms ease-out
   - 交互动画：300ms ease-in-out
   - 减少运动偏好支持

### 步骤 5：验证验收

**验收标准**：

| 指标 | 目标 | 验证方法 |
|------|------|----------|
| 视觉一致性 | 100% 使用 CSS 变量 | 搜索硬编码颜色 |
| 响应式完整性 | 6 断点全部正常 | 截图对比 |
| 暗黑模式 | 全组件支持 | 切换对比 |
| 可访问性 | WCAG AA | Lighthouse 检查 |
| 动画流畅度 | 60fps | DevTools 性能 |
