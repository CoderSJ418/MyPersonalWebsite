## 一、核心原则

### 1.1 分层管理原则

```
┌─────────────────────────────────────────────────────────────┐
│                    三层记忆架构                              │
├─────────────────────────────────────────────────────────────┤
│  全局层 (~/.iflow/AGENTS.md)                                │
│  ├── 用户身份、个人偏好                                      │
│  ├── iFlow 能力速查（简洁版）                                │
│  ├── 工作区概述                                              │
│  └── 通用 AI 协作要求                                        │
│                    ↓                                        │
│  项目层 (./AGENTS.md)                                        │
│  ├── 项目概述（精简）                                        │
│  ├── 技术栈摘要                                              │
│  ├── 项目特定约束                                            │
│  └── 通过 @ 导入详细模块                                     │
│                    ↓                                        │
│  模块层 (.iflow/*.md)                                        │
│  ├── iflow-capabilities.md → iFlow 能力详细版               │
│  ├── project-overview.md → 项目详细说明                      │
│  ├── tech-stack.md → 完整技术栈                              │
│  └── dev-standards.md → 开发规范                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 内容精简原则

| 层级 | 建议行数 | 核心内容 |
|------|----------|----------|
| 全局层 | 50-100 行 | 身份 + 能力速查 + 偏好 |
| 项目层 | 100-200 行 | 概述 + 导入 + 特定约束 |
| 模块层 | 不限 | 详细信息 |

### 1.3 模块化原则

**使用 `@` 导入语法**：

```markdown
# 项目主文件
@iflow/iflow-capabilities.md
@iflow/project-overview.md
@iflow/tech-stack.md
```

**优势**：
- 主文件简洁
- 模块可复用
- 易于维护
- 减少重复

## 四、常见错误与修正

### 4.1 错误：内容过长

❌ **错误做法**：
```markdown
# 项目记忆
[763行的完整项目文档...]
```

✅ **正确做法**：
```markdown
# 项目记忆
@iflow/project-overview.md
@iflow/tech-stack.md
@iflow/dev-standards.md
```

### 4.2 错误：信息混杂

❌ **错误做法**：
```markdown
# 项目记忆
项目概述...技术栈...开发规范...环境信息...Git提交规范...
```

✅ **正确做法**：
```markdown
# 项目记忆
## 项目概述
[只写概述]
## 技术栈
@tech-stack.md  # 详细内容导入
```

### 4.3 错误：缺少能力参考

❌ **错误做法**：
```markdown
# 项目记忆
项目信息...
<!-- AI 不知道自己有什么能力 -->
```

✅ **正确做法**：
```markdown
# 项目记忆
## iFlow 能力参考
@iflow/iflow-capabilities.md
## 项目概述
...
```

### 4.4 错误：全局记忆缺失

❌ **错误做法**：
```markdown
<!-- 全局记忆为空或缺失 -->
```

✅ **正确做法**：
```markdown
# 全局记忆
## 用户信息
称呼：杰哥
## iFlow 能力速查
[简洁的能力列表]
## AI 协作要求
称呼用户为"杰哥"
```

## 五、验证与调试

### 5.1 验证记忆是否生效

```bash
# 查看当前加载的记忆
/memory show

# 强制重新加载
/memory refresh

# 查看导入树结构
/memory show
```

### 5.2 常见问题排查

| 问题 | 排查步骤 |
|------|----------|
| AI 不知道能力 | 检查全局记忆是否有能力速查 |
| AI 忘记用户称呼 | 检查全局记忆是否有用户信息 |
| 项目信息不生效 | 检查 `/memory show` 是否加载 |
| 导入文件失败 | 检查路径、权限、循环引用 |

## 六、最佳实践总结

### 6.1 内容组织

| 原则 | 说明 |
|------|------|
| 分层管理 | 全局通用 → 项目特定 → 模块详细 |
| 模块化导入 | 大文件拆分为逻辑模块 |
| 版本控制 | 项目级纳入 Git，全局级不纳入 |

### 6.2 内容编写

| 原则 | 说明 |
|------|------|
| 自然语言 | 让 AI 容易理解 |
| 具体明确 | 给出具体示例 |
| 保持更新 | 及时更新配置 |

### 6.3 效果监控

| 原则 | 说明 |
|------|------|
| 定期清理 | 删除过时内容 |
| 观察效果 | 看 AI 行为是否符合预期 |
| 渐进优化 | 根据使用效果逐步完善 |
