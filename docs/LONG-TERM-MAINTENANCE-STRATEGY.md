## 一、执行摘要

本文档定义了个人项目的长期维护策略，确保在有限时间投入下保持项目可持续发展和高质量。

**核心原则：**
- 最小化维护负担，最大化自动化保障（Tight Scope + Safe Defaults）
- 文档即代码，与项目同步更新（Minimum Viable Documentation）
- 技术债务可视化，定期偿还（Runbook-Driven Ops）
- AI 辅助开发质量监控（AIUP - AI Unified Process）

**当前健康度快照（2026年3月1日）：**
| 维度 | 状态 | 详情 |
|------|------|------|
| 安全漏洞 | ⚠️ 需处理 | 4个漏洞（2中+2高）|
| 过期依赖 | ⚠️ 需更新 | 11个过期包 |
| 测试覆盖 | ❌ 不足 | 仅1个测试文件 |
| 代码注释 | ✅ 良好 | 无 TODO/FIXME 积压 |
| CI/CD | ✅ 正常 | lint + build 通过 |

## 四、决策框架

### 4.1 是否添加新功能的决策树

```
需要新功能？
    │
    ├─ 是否与项目核心定位相关？
    │       │
    │       ├─ 否 → 拒绝（避免范围蔓延）
    │       │
    │       └─ 是 → 继续
    │               │
    │               ├─ 能否用现有组件实现？
    │               │       │
    │               │       ├─ 是 → 复用/扩展
    │               │       │
    │               │       └─ 否 → 评估维护成本
    │               │               │
    │               │               ├─ 维护成本 > 开发成本 → 拒绝
    │               │               │
    │               │               └─ 维护成本 < 开发成本 → 批准
    │
    └─ 是否需要新依赖？
            │
            ├─ 是 → 评估依赖健康度
            │       │
            │       ├─ 月下载量 > 100K → 批准
            │       │
            │       └─ 月下载量 < 100K → 谨慎考虑
            │
            └─ 否 → 批准
```

### 4.2 是否删除代码/文档的决策树

```
考虑删除某项？
    │
    ├─ 是否被其他代码引用？
    │       │
    │       ├─ 是 → 先更新引用，再删除
    │       │
    │       └─ 否 → 继续
    │
    ├─ 是否有历史价值？
    │       │
    │       ├─ 是 → 移至 archive/ 或保留
    │       │
    │       └─ 否 → 继续
    │
    ├─ 是否最近 6 个月内使用过？
    │       │
    │       ├─ 是 → 保留
    │       │
    │       └─ 否 → 可以删除
    │
    └─ 删除后是否有回滚方案？
            │
            ├─ 有 Git 历史记录 → 可以删除
            │
            └─ 无备份 → 先备份再删除
```

### 4.3 依赖更新决策框架

| 更新类型 | 策略 | 风险评估 |
|----------|------|----------|
| Patch (x.x.Z) | 自动更新 | 低风险，直接合并 |
| Minor (x.Y.z) | 谨慎更新 | 中风险，测试后合并 |
| Major (X.y.z) | 计划更新 | 高风险，需要迁移计划 |
| 安全补丁 | 立即更新 | 优先级最高 |

## 五、记忆系统维护

### 5.1 AGENTS.md 更新策略

**触发条件：**
- 新增重要功能模块
- 技术栈重大变更
- 开发命令变化
- 项目结构重组
- 新增关键依赖

**更新频率：** 至少每月检查一次

**更新原则：**
- 保持简洁（目标 < 100 行）
- 突出核心信息
- 删除过时内容
- 同步实际状态

### 5.2 MCP 配置调整策略

**保留条件：**
- 最近 30 天内使用过
- 功能不可替代
- Token 开销合理

**禁用条件：**
- 与其他 MCP 功能重叠
- 超过 60 天未使用
- Token 开销过大且价值不高

**当前配置：**
| MCP | 状态 | 原因 |
|-----|------|------|
| context7 | ✅ 启用 | 实时文档查询 |
| playwright | ✅ 启用 | 浏览器测试 |
| memory | ❌ 禁用 | 功能重叠 |
| chrome-devtools | ❌ 禁用 | 功能重叠 |

### 5.3 知识积累方式

**代码注释：** 复杂逻辑必须有注释
**提交信息：** 遵循 Conventional Commits
**AGENTS.md：** 项目级知识
**README.md：** 用户级知识

## 六、代码质量保障

### 6.1 没有 E2E 测试的替代方案

**问题：** 当前只有 1 个单元测试文件，无 E2E 测试

**替代策略：**

| 层级 | 工具 | 覆盖范围 |
|------|------|----------|
| 静态检查 | ESLint + TypeScript | 代码规范、类型安全 |
| 单元测试 | Vitest | 核心工具函数 |
| 手动测试 | 测试清单 | 关键用户路径 |
| CI/CD | GitHub Actions | 每次提交验证 |

**手动测试清单（核心路径）：**

```
□ 首页加载正常
□ 导航跳转正常
□ 项目列表展示正常
□ 技能页面展示正常
□ 联系表单提交正常
□ 暗黑模式切换正常
□ 移动端响应式正常
```

### 6.2 CI/CD 流程增强建议

**当前 CI 流程：**
```yaml
lint → build → upload artifacts
```

**建议增强：**
```yaml
lint → build → test:run → upload artifacts → deploy (main only)
```

**具体改进：**
1. 添加测试步骤（即使覆盖率低）
2. 添加依赖安全审计（每周定时）
3. 添加构建大小检查（防止膨胀）

### 6.3 代码审查策略（单人开发）

**Self-Review 检查清单：**

```
□ 代码是否遵循项目风格？
□ 是否有未使用的导入/变量？
□ 是否有 TODO/FIXME 需要处理？
□ 是否更新了相关文档？
□ 提交信息是否清晰？
□ 是否影响了其他功能？
```

**AI 辅助审查：**
- 使用 iFlow CLI 进行代码审查
- 使用 BMAD workflows 进行架构评审
- 重点关注安全性和性能

## 八、技术债务管理

### 8.1 技术债务追踪

**追踪方式：**
- 使用 GitHub Issues 标签 `tech-debt`
- 在代码中使用 `// TODO:` 和 `// FIXME:`
- 定期扫描并记录到文档

**当前已知技术债务（2026年3月）：**
| 债务项 | 优先级 | 预计工作量 | 影响 |
|--------|--------|------------|------|
| 安全漏洞（4个） | 🔴 高 | 1小时 | 安全风险 |
| 测试覆盖率低 | 🟡 中 | 4-8小时 | 回归风险 |
| 主依赖版本过时 | 🟡 中 | 8-16小时 | 兼容性风险 |
| 无 E2E 测试 | 🟢 低 | 8-16小时 | 质量保障 |
| 文档可能过时 | 🟢 低 | 2-4小时 | 维护效率 |

### 8.2 债务偿还优先级矩阵

```
                    高影响
                      │
    安全漏洞          │         主依赖升级
    (立即处理)        │         (计划处理)
                      │
低紧急 ──────────────┼────────────── 高紧急
                      │
    文档更新          │         测试覆盖
    (有空处理)        │         (优先处理)
                      │
                    低影响
```

### 8.3 偿还时机

**立即偿还：**
- ✅ 安全漏洞（`npm audit fix`）
- ✅ 影响开发效率的问题
- ✅ 阻塞其他工作的问题

**计划偿还（本月）：**
- 测试覆盖率提升到 30%
- 小版本依赖更新

**计划偿还（本季度）：**
- Vue 3.5+ 升级评估
- Vite 7.x 升级评估
- Pinia 3.x 升级评估

**暂不偿还：**
- 不影响功能和性能的债务
- 计划重构的部分
- 即将废弃的代码

### 8.4 避免过度清理（重要）

**原则：**
- 不要在功能开发中途进行大规模清理
- 不要同时进行多个方向的优化
- 清理前确保有回滚方案（Git 分支）
- 清理后必须验证功能正常

**警惕信号：**
- 清理时间超过预期 2 倍 → 暂停评估
- 触发了意外的破坏性变更 → 回滚
- 影响了正在进行的其他工作 → 停止

**清理检查清单：**
```
□ 是否创建了备份分支？
□ 是否有回滚计划？
□ 清理范围是否明确？
□ 预计时间是否合理？
□ 是否会影响其他功能？
```

## 十、Solo Developer 最佳实践（研究总结）

### 10.1 核心模式

基于业界研究和个人项目特点，总结以下核心模式：

**Tight Scope（严格范围）**
- 只添加与核心定位相关的功能
- 拒绝范围蔓延（Scope Creep）
- 定期评估功能必要性

**Safe Defaults（安全默认）**
- 所有配置使用安全默认值
- 错误处理优于忽略
- 防御性编程

**Automated Checks（自动化检查）**
- CI/CD 必须包含 lint + build + test
- Git hooks 强制代码质量
- 定时安全审计

**Predictable Maintenance（可预测维护）**
- 固定的维护日历
- 明确的债务清单
- 量化的质量指标

### 10.2 AI 辅助开发注意事项

**AI 代码的风险：**
- 可能在长期维护中出问题
- 需要人工审查确保质量
- 建议使用 Spec-Driven Development

**AIUP（AI Unified Process）原则：**
1. 先写规格（Spec），再让 AI 实现
2. AI 实现后必须人工验证
3. 长期维护需要工程纪律
4. 不要依赖 "Vibe Coding"

### 10.3 Minimum Viable Documentation（MVD）

**MVD 原则：**
- 文档只保留最核心的内容
- 过时文档比没有文档更糟糕
- 文档应与代码同步更新

**MVD 核心文档清单：**
1. README.md - 项目入口
2. AGENTS.md - AI 记忆
3. package.json - 依赖配置
4. CI/CD 配置 - 自动化流程

## 十一、总结

### 核心策略

1. **自动化优先** - 让机器做重复的事
2. **文档最小化** - 只保留必要文档
3. **技术债务可视化** - 追踪并定期偿还
4. **质量指标量化** - 用数据说话
